## 🚨 ПРАВИЛО ПЕРЕД MERGE В MAIN

**Перед каждым `git push origin main` после значимой фичи (не hotfix-1-строчник) Claude обязан САМ запустить локальный code review через subagent `feature-dev:code-reviewer`**.

Это правило проекта: безопасность production-системы важнее экономии 1-2 минут. На практике (этап 0c, 2026-05-19) ревью находил баги уровня «миграция n8n с Railway сломается, бэкапы будут уничтожать сами себя».

### История правила

Изначально (2026-05-18) правило требовало пользователю запускать плагин `/ultrareview` (многоагентный cloud-обзор, платный). С 2026-05-19 заменено на локальный `feature-dev:code-reviewer` потому что `/ultrareview` недоступен в текущей среде Claude Code (`"isn't available in this environment"`). Локальный агент бесплатный, фокусируется на high-priority issues с confidence-based filtering.

### Когда применяется

- ✅ Завершение этапа (0, 1, 2, ...) или большой части этапа
- ✅ Изменения в RLS политиках, миграциях БД, auth, n8n workflows
- ✅ Изменения в `calculator.browser.js` или его TypeScript-аналоге
- ✅ Изменения в `infra/` (docker-compose, Caddyfile, backup скрипты)
- ❌ Однострочные правки текста, опечатки, типов, обновления документации

### Как применять

1. **Сразу после commit, перед push**: Claude вызывает `feature-dev:code-reviewer` через Agent tool. В промпте описывает:
   - **Контекст** проекта (что это, какой стек, кто заказчик)
   - **Файлы для ревью** (конкретные пути из этого commit'а)
   - **На что обратить особое внимание** (безопасность, корректность, edge cases для конкретной фичи)
   - **Формат ответа**: только high-priority issues с уверенностью 80+, конкретные фиксы, лимит ~600-800 слов
2. **Показать результат пользователю** простыми словами (без англицизмов), объяснить каждый найденный баг и его последствия
3. **Спросить через AskUserQuestion**: применить все фиксы / только критичные / пропустить
4. **После применения фиксов** — push в main. Если фиксы существенные — допустим повторный прогон ревью

### Что НЕ заменяет ревью

- Vitest unit-тесты — не заменяют (там code-level baгs, не функциональные)
- TypeScript type-check — не заменяет (там логические ошибки)
- Линтер — не заменяет (там runtime ошибки)

Только ревью смотрит на **смысл** изменений и ловит баги типа «эта команда работает не как ты думаешь».

### Опционально: /ultrareview если станет доступен

Если когда-нибудь плагин `/ultrareview` (cloud, платный) станет доступен в среде — можно дополнительно запустить его пользователем для второго мнения. Это **не обязательно** — локальный ревью самодостаточен.

## Архитектура кода (index.html)

### Структура файла
Монолитный HTML файл разделён на логические секции:
1. **HTML разметка** (строки 1-500): интерфейс с drag-and-drop зоной, таблицей позиций
2. **CSS стили** (строки 8-250): Tailwind + кастомные стили для drag-drop, таблиц, статусов
3. **JavaScript логика** (строки 500-3000+): state management, парсинг, расчёты, API интеграция

### Ключевые константы типов металла (~строки 900-980)

```javascript
const TYPES_WITHOUT_PIECES = [...]  // Металлы без поля "штуки" (канаты, проволока, профнастил)
const FASTENER_TYPES = [...]        // Крепёж (только вес и количество, без длины)
const SHEET_TYPES = [...]           // Листовые типы (ширина/длина листа вместо длины)
const TYPES_WITH_STANDARDS = [...]  // Типы с выбором стандарта (профнастил, рельсы, листы ПВ/рифлёные/окраш)
const TYPES_WITHOUT_STEEL = [...]   // Типы без выбора марки стали (база уже содержит коэффициенты)
const TYPES_WITH_LENGTH_PIECE = [...] // Типы с полем "Длина 1 шт"
```

Эти константы определяют, какие поля показывать в интерфейсе для каждого типа металла.

### Ключевые функции

**Получение стандартов/вариантов** (~строка 1167):
```javascript
function getStandardsForMetal(metalKey, size)
```
Возвращает массив доступных стандартов для металла (например, для профнастила НС35: разные толщины по ГОСТ 24045).

**Рендеринг строки таблицы** (~строка 1600):
```javascript
function renderRow(item, index)
```
Генерирует HTML строки таблицы с полями в зависимости от типа металла. Использует `getFieldsConfig()` для определения видимости полей.

**Обработчики изменений** (~строки 1790-1850):
```javascript
function onSizeChange(index, value)    // Изменение размера -> пересчёт
function onFieldChange(index, field, value)  // Изменение любого поля
function onSteelChange(index, value)   // Изменение марки стали -> пересчёт
function onStandardChange(index, value) // Изменение стандарта -> пересчёт
```

**Пересчёт строки** (~строка 1980):
```javascript
function recalculateRow(index, changedField)
```
Центральная функция пересчёта. Собирает параметры (size, steel, standard) и вызывает `calculateMetal()`. Обрабатывает результаты с учётом разных форматов ответа (объект/число/результат с actual).

### Передача стандарта в calculateMetal

В 4 местах кода стандарт передаётся в библиотеку расчётов:
```javascript
params.standard = standardObj;  // {name: "...", coefficient: X.XX}
```

Искать по строке `params.standard = standardObj` (приблизительно строки 1895, 2005, 2295, 2835).

### Конфигурация полей

```javascript
function getFieldsConfig(metalKey) // ~строка 1240
```

Возвращает объект с флагами:
- `isSheet` - листовой тип (показывать ширину/длину)
- `isFastener` - крепёж (скрыть длину)
- `noPieces` - скрыть штуки
- `hasStandards` - показать выбор стандарта
- `noSteel` - скрыть марку стали
- `hasLengthPiece` - показать "Длина 1 шт"

## Типы металлов и их особенности

### Листовые типы (SHEET_TYPES)
- Имеют поля "Ширина" и "Длина" вместо обычного "Метры"
- Поле "Метры" показывает площадь в м²
- Примеры: листы г/к, х/к, оцинк, окрашенные, рифлёные, ПВ

### Типы со стандартами (TYPES_WITH_STANDARDS)
- Профнастил (окраш/оц): выбор типа и толщины (НС35 0.6х1000, 0.7х1000 и т.д.)
- Рельсы: выбор типа и стандарта (ГОСТ, EN, AREMA)
- Лист ПВ/ПВ оц: выбор ТУ стандарта
- Лист рифлёный: выбор типа рифления (чечевица/ромб)
- Лист окрашенный: выбор типа оцинковки (I класс, II класс)

### Типы без марки стали (TYPES_WITHOUT_STEEL)
База данных уже содержит все нужные коэффициенты, марка стали не влияет на расчёт. Примеры: профнастил, шпунты, рельсы.

### Оцинкованные типы
При расчёте веса применяется коэффициент оцинковки 1.03 (дополнительный вес цинкового покрытия). Коэффициент применяется в функции `recalculateRow`.

### Формулы расчёта

Формулы определяются в базе данных metals.json. Некоторые типы используют специальные форматы:
- `profnastil_area` - расчёт по площади с коэффициентом стандарта
- `rels_linear` - линейный вес рельса с вариантами по стандарту
- `shpynt_3fields` - шпунты с тремя полями ввода (тип/размер/метод)

## Отладка

### Консольные логи
Код содержит множество `console.log` с эмодзи-префиксами:
- 📋 - парсинг входных данных
- 🔄 - пересчёт строки
- 📊 - результат расчёта
- ❌ - ошибка
- ✅ - успешная операция
- 🔍 - отладочная информация

Открыть Chrome DevTools (F12) → Console для отслеживания.

### Частые проблемы

1. **Вес не пересчитывается при смене размера/марки**
   - Проверить, что `onSizeChange`/`onSteelChange` вызывает `recalculateRow(index)`

2. **Стандарт не применяется**
   - Проверить передачу `params.standard = standardObj` во всех 4 местах вызова `calculateMetal`
   - Стандарт должен быть объектом `{name, coefficient}`, не строкой

3. **Поля не показываются/не скрываются**
   - Проверить константы типов (SHEET_TYPES, FASTENER_TYPES и т.д.)
   - Проверить `getFieldsConfig()` для данного типа металла

4. **Неправильный ключ типа металла**
   - Пример: `sheet_riffle` → `sheet_checkered`
   - Сверить с metals.json из metal-calculator

## API интеграция с n8n workflows

Frontend взаимодействует с backend через n8n webhooks, развёрнутые на Railway.

### Базовый URL
```
https://primary-production-4e88.up.railway.app/webhook
```

### Endpoints

**1. Парсинг файлов** (`/calc-parse`)
- **Метод**: POST
- **Назначение**: Распознавание Excel, PDF, изображений
- **Запрос**: multipart/form-data с файлом
- **Ответ**: массив распознанных позиций металла

**2. Формирование запроса поставщикам** (`/supplier-request`)
- **Метод**: POST
- **Назначение**: Создание Excel с позициями, сохранение в БД
- **Запрос**: JSON с массивом позиций
- **Ответ**:
  ```json
  {
    "excel_base64": "...",
    "filename": "Запрос_поставщикам_YYYY-MM-DD.xlsx"
  }
  ```

**3. Отправка поставщикам** (`/send-supplier-request`)
- **Метод**: POST
- **Назначение**: Рассылка email поставщикам через Resend API
- **Запрос**:
  ```json
  {
    "request_id": "...",
    "category": "nn|moscow|factories",
    "items": [...],
    "totals": {...}
  }
  ```
- **Ответ**: статус отправки

**4. Получение сводки** (`/get-supplier-summary?request_id=...`)
- **Метод**: GET
- **Назначение**: Получить сводку лучших предложений от поставщиков
- **Ответ**: Excel файл со сравнением цен

### Backend стек (через n8n workflows)

- **Database**: Supabase (хранение запросов, позиций, ответов поставщиков)
- **Email**: Resend API (рассылка запросов поставщикам)
- **Хостинг**: Railway (n8n workflows)
- **Формирование Excel**: ExcelJS (внутри n8n workflows)
